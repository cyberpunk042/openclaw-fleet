# Agent Directory Cleanup — Templates vs Runtime

**Date:** 2026-04-01
**Status:** PLANNING — restructure agent directories for clean IaC
**Blocker level:** Should be done BEFORE writing new CLAUDE.md/HEARTBEAT.md

---

## 1. The Problem

The agents/ directory mixes committed and generated files with no
clear logic:

```
CURRENT MESS:
  agents/architect/
    CLAUDE.md        ← committed (source of truth? or template output?)
    HEARTBEAT.md     ← committed (same question)
    agent.yaml       ← committed (config or runtime?)
    context/
      fleet-context.md  ← committed but OVERWRITTEN every 30s by brain
    IDENTITY.md      ← gitignored at root, but TRACKED for persistent agents
    SOUL.md          ← same inconsistency
    TOOLS.md         ← gitignored (generated by gateway)

  agents/devsecops-expert/
    .gitignore       ← OVERRIDES root gitignore to track IDENTITY/SOUL/etc
    IDENTITY.md      ← committed (because of local .gitignore override)
    SOUL.md          ← committed
    AGENTS.md        ← committed
    CLAUDE.md        ← committed (this one is actually good — Cyberpunk-Zero)
```

Problems:
1. **context/ files committed but overwritten every 30s** — git noise
2. **Inconsistent gitignore** — root says ignore IDENTITY.md, 3 agents override
3. **No clear boundary** — which files are "source" vs "output"?
4. **CLAUDE.md is both source AND needs rewriting** — confusing
5. **Agent directories are treated as both IaC and runtime workspace**

## 2. The Clean Model

### Principle: Templates + Config = Source. Agent Directories = Runtime Output.

```
COMMITTED (IaC source of truth):
  agents/_template/              ← ALL templates
    agent.yaml.template          ← with {AGENT_NAME}, {ROLE}, {MODEL} placeholders
    CLAUDE.md/                   ← per-role CLAUDE.md files (10 of them)
      architect.md
      software-engineer.md
      qa-engineer.md
      devops.md
      devsecops-expert.md
      fleet-ops.md
      project-manager.md
      technical-writer.md
      ux-designer.md
      accountability-generator.md
    HEARTBEAT.md/                ← per-role or template
      driver-pm.md
      driver-fleet-ops.md
      driver-devsecops.md
      worker.md                  ← shared worker template
    IDENTITY.md.template         ← with {AGENT_NAME}, {DISPLAY_NAME}, {ROLE}
    SOUL.md.template             ← shared values + {ROLE_VALUES} placeholder
    AGENTS.md.template           ← with {SYNERGY_POINTS} from config
    MC_WORKFLOW.md               ← shared workflow instructions
    STANDARDS.md                 ← shared standards
    MC_API_REFERENCE.md          ← tool reference
    mcp.json.template            ← with {AGENT_NAME}, per-role servers from config
    .claude/settings.json        ← permissions

  config/
    agent-identities.yaml        ← names, display names, roles
    agent-tooling.yaml           ← per-role MCP servers, plugins, skills
    agent-autonomy.yaml          ← per-role thresholds, wake triggers
    fleet.yaml                   ← fleet-wide config
    phases.yaml                  ← delivery phases
    skill-assignments.yaml       ← skills per agent

  scripts/
    provision-agents.sh          ← generates agent directories from templates + config
    push-framework.sh            ← pushes shared files to agent workspaces

RUNTIME (fully untracked via .gitignore):
  agents/architect/              ← ENTIRE directory gitignored
  agents/software-engineer/
  agents/qa-engineer/
  agents/devops/
  agents/devsecops-expert/
  agents/fleet-ops/
  agents/project-manager/
  agents/technical-writer/
  agents/ux-designer/
  agents/accountability-generator/
```

### .gitignore Change

```gitignore
# Agent runtime directories — generated from templates
# Only _template/ and config/ are source of truth
agents/*/
!agents/_template/
```

This is ONE line that replaces the current mess of per-file and
per-agent gitignore rules.

## 3. Provisioning Flow

```
make provision
  ↓
scripts/provision-agents.sh
  ↓
For each agent defined in config/agent-identities.yaml:
  ↓
  1. Create agents/{name}/ directory (if not exists)
  
  2. Generate agent.yaml from template:
     Fill: {AGENT_NAME}, {DISPLAY_NAME}, {ROLE}, {MODEL},
           {CAPABILITIES}, {CONSTRAINTS}
     Source: config/agent-identities.yaml
  
  3. Copy CLAUDE.md for this role:
     Source: agents/_template/CLAUDE.md/{role}.md
     → agents/{name}/CLAUDE.md
  
  4. Copy HEARTBEAT.md for this role:
     Source: agents/_template/HEARTBEAT.md/{role-type}.md
     → agents/{name}/HEARTBEAT.md
  
  5. Generate IDENTITY.md from template:
     Fill: {AGENT_NAME}, {DISPLAY_NAME}, {FLEET_ID}, {ROLE}
     Source: agents/_template/IDENTITY.md.template + config
  
  6. Generate SOUL.md from template:
     Fill: {ROLE_VALUES} (per-role values section)
     Source: agents/_template/SOUL.md.template
  
  7. Generate AGENTS.md from template:
     Fill: synergy points from fleet-elevation/15 contribution matrix
     Source: agents/_template/AGENTS.md.template + config
  
  8. Generate mcp.json from template + tooling config:
     Fill: fleet MCP server + role-specific servers from agent-tooling.yaml
     Source: agents/_template/mcp.json.template + config/agent-tooling.yaml
  
  9. Copy shared files:
     MC_WORKFLOW.md, STANDARDS.md, MC_API_REFERENCE.md
     .claude/settings.json
  
  10. Create empty context/ directory (brain writes here)
  
  11. Install plugins per config:
      Claude-Mem (all), Context7 (architect+engineer), etc.

Result: agents/{name}/ is a complete, ready-to-run workspace.
All generated from committed templates + config.
Reproducible: `make provision` from clean state = identical output.
```

and treat it like any other fleet module with tests.

## 5. Memory Sync Strategy

The PO said: "whatever we would want to sync back from their memory
we would strategically do it."

Agent MEMORY.md and daily logs exist at runtime. They're valuable
but NOT committed. Strategic sync means:

```
Agent produces valuable learning during runtime:
  "Discovered that Radix Select needs aria-label for accessibility"
  ↓
This goes into agent's local MEMORY.md (auto-memory)
  ↓
If strategically valuable for the fleet:
  Agent posts to board memory: fleet_chat(
    "Architecture decision: Radix Select requires aria-label",
    tags=["architecture", "decision", "ux"]
  )
  ↓
  Board memory is searchable by ALL agents
  ↓
If permanently valuable:
  Ingest into RAG knowledge base (when built)
  → SQLite-backed, survives docker purge
  → Available to all agents on future tasks
```

What we DON'T do:
- Auto-commit agent MEMORY.md to git (noise, personal to agent)
- Sync agent state back to templates (one-way: templates → agents)
- Treat agent runtime as source of truth (templates + config are truth)

## 6. Pre-Embedding and Dynamic Content

The brain writes `context/fleet-context.md` and `context/task-context.md`
every 30 seconds. These are ALWAYS runtime — never committed, never
templated. They're the bridge between orchestrator and agent.

```
Orchestrator (30s cycle)
  ├── Reads: MC API (tasks, agents, memory)
  ├── Computes: role-specific data (providers)
  ├── Formats: preembed (structured markdown, FULL data)
  └── Writes: agents/{name}/context/fleet-context.md
              agents/{name}/context/task-context.md
  ↓
Gateway reads context/ when agent heartbeats
  ↓
Agent wakes with FULL awareness (zero MCP calls for awareness)
```

This doesn't change with the cleanup. Context/ files stay runtime.
They're just now CLEARLY runtime (entire agents/ directory is untracked).

## 7. Implementation Steps

### Step 1: Create Template Structure

```bash
mkdir -p agents/_template/CLAUDE.md
mkdir -p agents/_template/HEARTBEAT.md

# Move current CLAUDE.md files to templates
for agent in architect software-engineer qa-engineer devops \
             devsecops-expert fleet-ops project-manager \
             technical-writer ux-designer accountability-generator; do

# Create HEARTBEAT.md templates
```

## 8. Order of Operations

This cleanup should happen BEFORE writing new CLAUDE.md/HEARTBEAT.md files:

1. **Create template structure** (Step 1)
2. **Write provision script** (Step 2)
3. **Update .gitignore** (Step 3)
4. **Remove committed agent files** (Step 4)
5. **Move accountability src/** (Step 5)
6. **Test** (Step 6)
7. THEN write new CLAUDE.md per fleet-elevation specs (into _template/)
8. THEN write new HEARTBEAT.md per role (into _template/)
9. Run `make provision` → generates all agent directories
10. Live test with properly provisioned agents

This ensures: new CLAUDE.md files go into the RIGHT place (_template/),
are generated into agent directories via IaC, and are reproducible.
